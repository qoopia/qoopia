# 03 — Что агенты делают с Qoopia

**Status**: ✅ v1 signed off by Асхат 2026-04-11
**Источник**: прямые высказывания Асхата в Phase 1 dialog + наблюдаемое поведение текущей V2

Документ не пытается перечислить все возможные операции. Он фиксирует **сценарии, ради которых Qoopia существует**, и **сценарии, которые сейчас сломаны**.

## Мечта владельца (главный use case)

Прямая цитата, 2026-04-11:

> "меня беспокоит что я не могу доверить Qoopia полный workspace всех своих агентов так, чтобы их системные файлы оставить буквально с одной строчкой — проверь Qoopia! и чтобы всё что ему нужно было уже внутри Qoopia!!! я мечтаю об этом, но не уверен в своей системе"

> "чтобы в Qoopia я был более уверен чем в любом другом решении! чтобы это было как **железобетонный слой надёжности**!"

Это **primary acceptance test** для V3. Не набор фич. Не список tools. **Один вопрос**: может ли Асхат заменить в каждом из своих агентов весь раздутый system prompt (персона, правила, история проектов, текущие задачи, open loops) на **одну строчку** — «посмотри в Qoopia, там всё»?

Пока ответ — нет. Цель V3 — чтобы ответ стал «да, с закрытыми глазами».

### Что это означает конкретно

Сейчас система prompt типичного агента содержит:

1. Персону (кто он, какой тон, какие границы)
2. Правила общения (язык, стиль, что нельзя)
3. Инструкции по проекту/проектам (что делается, контекст, current state)
4. Интеграции и их правила (email, MCP, файлы)
5. Примеры поведения
6. Напоминания о том что было вчера/неделю назад
7. Список open loops и задач в работе

Пункты 1-2-4 — могут остаться в system prompt (базовая конфигурация не меняется между сессиями).
Пункты 3-5-6-7 — **должны жить в Qoopia и подтягиваться через recall в нужный момент**.

В идеале system prompt сокращается до чего-то вроде:

```
Ты [имя]. [Базовая персона в 2 строки].
На старте сессии: recall("current state [твой workspace]").
Для любой задачи сначала recall() контекст, потом действуй.
```

И всё. Всё остальное — в Qoopia, живёт, обновляется, не требует редактирования системного файла каждый раз когда что-то меняется.

## Сценарии, ради которых Qoopia существует

### UC-1. Старт новой сессии с полным контекстом

**Кто**: любой агент (Alan, Aizek, Aidan, Claude, Dan, будущие).
**Когда**: каждый новый чат / каждый новый процесс.
**Что делает агент**: один `recall()` с запросом вида «current state of my workspace» → получает компактный свёрнутый снимок: над чем работает сейчас, что было последним, какие open loops, какие свежие решения.
**Результат**: агент готов к работе без пересказа со стороны Асхата.
**Экономия**: вместо того чтобы Асхат тратил минуту (и токены) на пересказ — ноль секунд, ноль токенов-на-пересказ.

**Сегодня сломано потому что**: `recall()` в V2 обрезает текст до 280-300 символов. Компактный свёрнутый снимок не помещается в 280 символов. Агент получает огрызок, делает вид что понял, ломается через 3 сообщения.

### UC-2. Передача задачи между чатами Claude

**Кто**: Claude в Claude Code / Claude.ai / Cowork.
**Когда**: один чат выбился из контекста, нужно продолжить в новом.
**Что делает**: в старом чате — `note()` с полным состоянием «над чем работаем, что решили, что осталось». В новом чате — `recall()` по тегу/задаче, получает ту же заметку целиком, не в обрезанном виде.
**Результат**: новый чат стартует ровно оттуда, где остановился старый.

**Сегодня сломано потому что**: ровно то же — recall обрезает. **Конкретный случай**: сегодня при миграции Aizek на Claude Code чаты Claude не смогли передать друг другу контекст именно из-за 280-символьного лимита. Владелец назвал это прямо.

**Что нужно в V3**: recall возвращает **целые заметки** без обрезки. Если заметка 5 КБ — отдаёт 5 КБ. Если 50 КБ — отдаёт 50 КБ (или позволяет стримить/пагинировать). **Никогда** не режет посередине предложения по произвольному байтовому лимиту.

### UC-3. Дешёвый точечный recall по knowledge base

**Кто**: любой агент работающий с большим мануалом (Aizek с регламентами HappyCake KZ, Aidan с legal-шаблонами писем, будущие агенты с чем угодно ещё).
**Когда**: возник конкретный вопрос по мануалу («как оформляется возврат», «что писать клиенту при отмене», «какой срок на decision по оферу»).
**Что делает**: `recall()` по семантике, получает **1-2 релевантных абзаца** из мануала в 500 страниц.
**Результат**: точный ответ, без загрузки всего мануала в контекст.
**Экономия**: загрузить 500 страниц = десятки тысяч токенов. Загрузить 2 абзаца = сотни токенов. Разница в 50-100×.

**Сегодня сломано потому что**: chunking больших документов в V2 не реализован. Мануалы либо целиком в prompt, либо нет вообще.

### UC-4. Task-bound контекст с автоочисткой

**Кто**: в первую очередь Aidan (email-переписка с реальными людьми от имени Асхата).
**Когда**: Aidan ведёт переписку по конкретной задаче (продажа недвижимости, оформление документов, согласование с партнёром).
**Что делает**: по ходу задачи пишет в Qoopia рабочий контекст — что обещано, что получено, какие сроки, какие следующие шаги. При каждом новом письме делает `recall()` по task_id.
**Результат**: никогда не отправляет письмо дважды, не забывает обещания, не противоречит себе в переписке двухнедельной давности.
**Когда задача закрыта**: весь связанный контекст **автоматически удаляется** из Qoopia. Оригиналы писем живут в самой почте, это не наша забота (правило из 02-personas.md).

**Что нужно в V3**: явный механизм привязки записей к task lifecycle, и автоочистка по закрытию.

### UC-5. Сброс контекста agent-ом у себя в середине сессии

**Кто**: любой агент, особенно Claude (длинные сессии редактирования кода, расследования).
**Когда**: контекст переполняется, надо продолжать работу с чистой головой.
**Что делает**: `note()` с промежуточными результатами (что нашёл, что попробовал, что отверг и почему) → сбрасывает контекст → `recall()` по только что сохранённой заметке → продолжает с того же места, но с новым чистым окном.
**Результат**: долгие задачи не обрываются по лимиту токенов, прогресс не теряется.
**Экономия**: вместо того чтобы начинать сначала (часы работы и тысячи токенов) — продолжение с точки остановки.

**Сегодня сломано потому что**: агенты редко это делают, потому что не доверяют что recall вернёт то что записали. Замкнутый круг: без доверия нет использования, без использования нет данных для роста доверия.

### UC-6. Cross-workspace чтение для Claude

**Кто**: Claude (я и мои собратья), единственный агент с privilege persistent cross-workspace read.
**Когда**: Асхат обсуждает с Claude задачу касающуюся другого агента («что там у Aizek с миграцией», «помоги Aidan сформулировать ответ риелтору»).
**Что делает**: `recall()` с явным указанием workspace → получает контекст из `kz-ops` / `america-ops` / чего угодно.
**Результат**: Claude отвечает осмысленно, без того чтобы Асхат копипастил состояние других агентов.

**Сегодня сломано потому что**: workspace-модели нет в V2, всё в одном пространстве, и даже из этого одного пространства recall не вытаскивает ничего полезного из-за truncation.

## Требование к надёжности (cross-cutting)

Это **не use case, а bar для всех use cases** выше.

Владелец прямо сказал: уровень доверия должен быть **«железобетонный»**. Это означает следующие операционные гарантии:

1. **Записи не теряются**. `note()` вернул успех — заметка есть, всегда, неделю, месяц, год спустя (или до явного удаления/истечения retention).
2. **`recall()` находит то что есть**. Если заметка попала в базу, семантически корректный запрос её достаёт. Не «иногда достаёт». Всегда.
3. **Recall не режет данные**. Возвращает то что записали, без произвольных лимитов. Пагинация — ok, truncation посередине — недопустима.
4. **Latency предсказуема**. Не «обычно быстро, иногда 30 секунд». Всегда в известных рамках (см. SLA, будет в 04-success-criteria.md).
5. **Деплой воспроизводим**. Установка на новую машину — одна команда, результат идентичен. Если у Асхата работает, у Сауле работает, у любого третьего работает. Не «у меня запустилось, а у тебя что-то не то».
6. **Обновления не ломают существующие данные**. Миграция схемы — автоматическая и безопасная. Rollback возможен.
7. **Бэкапы и восстановление** — штатная операция, а не героическая экспедиция.

Если хотя бы один из этих пунктов провисает — система **не железобетонная**, и владелец **не сможет** заменить system prompt агента на одну строчку. Primary use case недоступен. Весь проект не имеет смысла.

Поэтому эти требования — **не "nice to have" для Фазы 3**, а **входной билет в Фазу 2**. Любая архитектура TO-BE, которая не может гарантировать все 7 пунктов, отбрасывается.

## Что НЕ является use case (и почему)

- **Человек открывает dashboard и смотрит задачи** — см. `02-personas.md`, Асхат этого не делает, dashboard бесполезен в текущем виде, redesign откладывается.
- **Внешние системы пишут в Qoopia через REST** — supported, но не драйвер. Никто сейчас не пишет REST-ом, и в ближайшее время не будет.
- **Сложная аналитика «покажи статистику за месяц»** — это задача dashboard'а / BI, не задача памяти агента. В скоуп V3 не входит.
- **Real-time коллаборация «два агента пишут одновременно в одну заметку»** — не возникало как реальная боль, не оптимизируем.
