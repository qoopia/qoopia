# 05 — Чем Qoopia V3 НЕ должна быть

**Status**: ✅ v2 signed off by Асхат 2026-04-11 (after Simplicity Pass)
**Источник**: производные от `01–04`, явные отказы Асхата в Phase 1 dialog, и жёсткие отсечения которые нужны чтобы проект не расползся.

Без этого документа любая реализация со временем превращается в «всё для всех». Здесь фиксируется чем Qoopia **сознательно** не является. Каждый пункт — отказ от соблазна, который реален.

## Принцип отсечения

Если фича/сценарий/интеграция не помогает **primary acceptance test** (заменить system prompt агента на «проверь Qoopia» с железобетонным доверием) — она не в scope V3. Точка. Возможно в V4, V5. Не сейчас.

## NG-1. Qoopia — не операционная CRM для человека

Что это значит:
- **Не** инструмент для того чтобы Асхат (или любой другой человек) руками заводил задачи, листал deal pipeline, отмечал галочки выполнения
- **Не** замена Trello / Notion / Linear / Asana
- **Не** система где человек — primary user

Почему: `01-why.md` и `02-personas.md` однозначно установили что primary user — **агенты**. Человек взаимодействует через агентов.

Что это НЕ запрещает: entity-типы `task`, `deal`, `contact`, `finance` остаются в Qoopia как **reference data** для агентов. Агент может создать/обновить задачу по ходу работы. Но это инструмент агента, а не человеческий task manager.

## NG-2. Qoopia — не BI / аналитика / reporting

Что это значит:
- **Не** строит дашборды «сколько задач закрыто за месяц»
- **Не** генерирует отчёты по продажам / финансам / метрикам бизнеса
- **Не** визуализирует временные ряды, не рисует графики
- **Не** отвечает на вопросы типа «покажи все задачи со сроком в июне где assignee = Aizek и status = waiting»

Почему: это другой класс задач с другим tradeoff'ом (batch-pipeline вместо low-latency retrieval, analytical SQL вместо семантический поиск, dashboard UI вместо MCP tools). Попытка делать оба — гарантированный провал обоих.

Что это НЕ запрещает: Qoopia через REST API может отдавать сырые данные, на основе которых **отдельная BI-система** (не часть V3) будет строить отчёты. Сейчас этого нет и не приоритет.

## NG-3. Qoopia — не message bus / очередь задач / оркестратор агентов

Что это значит:
- **Не** механизм «передай задачу от Alan к Mia»
- **Не** очередь pending jobs для агентов
- **Не** система которая решает кого разбудить когда наступило событие
- **Не** workflow engine

Почему: это отдельный класс систем (Temporal, Airflow, Redis pub/sub, message brokers). У них своя семантика, свои гарантии, свой reliability model. Qoopia — **память**, а не **передача**.

Что это НЕ запрещает: агент может **прочитать** в Qoopia что другой агент туда положил («Alan оставил заметку что передал SMM-задачу Mia»). Но это чтение статуса, а не акт передачи. Передача — через внешний механизм (прямое обращение, внешний messenger, Telegram).

## NG-4. Qoopia — не эмбеддинг-провайдер / не обучение моделей

Что это значит:
- **Не** тренирует собственные embedding-модели
- **Не** file-tune'ит LLM
- **Не** хостит модели

Почему: это отдельная инфраструктура. Qoopia **использует** существующие модели (OpenAI API, Voyage, или локальные через onnx/ollama — решим в Фазе 3), но не занимается их производством.

## NG-5. Qoopia — не end-user приложение с UI

Что это значит:
- **Не** имеет собственного веб-интерфейса для обычных пользователей
- **Не** является тем что Асхат или Сауле открывают в браузере чтобы «поработать»

Dashboard из V2 — отдельная история. Он существует, считается бесполезным в текущем виде, и его **redesign откладывается** до момента когда агентская часть работает железобетонно. Это зафиксировано в `01-why.md` и `02-personas.md`.

Практически в V3: если и будет какой-то UI, то:
- **Operator UI** для админа (смотреть workspace'ы, подключать/отключать агентов, читать логи) — в scope, минимальный
- **End-user dashboard** — вне scope V3, redesign в отдельной фазе

## NG-6. Qoopia — не универсальное хранилище файлов

Что это значит:
- **Не** Dropbox / Google Drive / S3-заменитель
- **Не** хранилище бинарников, картинок, видео, PDF как файлов
- **Не** CDN

Почему: для этого есть специализированные решения, и они уже неплохо работают. Qoopia фокусируется на **текстовой** памяти и knowledge.

Что это НЕ запрещает: Qoopia может индексировать **содержимое** PDF/DOCX (извлечённый текст) и хранить ссылку на оригинал. Сам файл живёт где-то ещё.

## NG-7. Qoopia — не проксирует вызовы внешних сервисов

Что это значит:
- **Не** является обёрткой «агент сказал Qoopia найти в Gmail, Qoopia сходила в Gmail и вернула»
- **Не** wrapper над email / calendar / Slack / HappyCake Apps

Почему: это были бы MCP-tool'ы другого типа (интеграции), не память. Смешивать нельзя — при миксе получается непонятный монстр который и не память, и не интеграция.

Что это НЕ запрещает: **результаты** работы внешних интеграций (другой MCP) могут попадать в Qoopia через `note()`. Это как раз правильно. Aidan сходил в Gmail через Gmail MCP, получил письмо, сбросил краткое содержание в Qoopia через `note()` — отлично.

## NG-8. Qoopia — не security/identity provider

Что это значит:
- **Не** хранит пароли
- **Не** является SSO
- **Не** выдаёт OAuth-токены внешним системам
- **Не** решает «имеет ли пользователь X право делать Y в системе Z»

Почему: это отдельный домен со своими стандартами (OAuth, SAML, OIDC). Qoopia имеет свою модель workspace/agent isolation (см. `02-personas.md`), и она касается **только доступа внутри Qoopia**. Для «кто этот пользователь» — внешний identity provider.

## NG-9. Qoopia не оптимизируется под «случайного посетителя»

Явная формулировка владельца (2026-04-11):

> «это решение должно быть ответом на реальную боль! оно не подойдёт тому, кто не ищет конкретного решения своей боли!»

Что это значит:
- **Не** нужен friendly-onboarding в стиле «запусти за 2 минуты и понрави мне фичами»
- **Не** нужны демо-данные, примеры «как если бы у вас были агенты»
- **Не** нужны завлекательные маркетинговые сценарии
- **Не** нужно снижать порог входа ниже уровня «человек реально запустил несколько агентов»

Что это НЕ запрещает: документация должна быть отличной. Установка должна быть одной командой. Но это для тех **кто уже понял зачем Qoopia**, а не для привлечения прохожих.

## NG-10. Qoopia не пытается быть «единственным» для всех задач

Что это значит:
- **Не** заменяет собой все остальные tool'ы в стеке агента
- **Не** является «вот тебе Qoopia, больше ничего не нужно»

Агент по-прежнему пользуется:
- Gmail MCP — для работы с почтой
- Calendar MCP — для календаря
- File system / code editing tools — для кода
- Специализированными MCP для своего домена (HappyCake Apps в будущем для Aizek, и т.д.)

Qoopia — **слой памяти и знаний поверх всего этого**. Не центр вселенной.

## NG-11. V3 не мигрирует 100% содержимого V2 автоматически

Что это значит:
- **Не** тратим недели на то чтобы перетащить каждую заметку V2 в идеальную структуру V3
- Основной массив V2 уходит в специальный workspace `legacy` (read-only) при миграции (решено в `02-personas.md`)
- Дальше агенты по мере работы перекладывают нужное в правильные workspace'ы
- Через N месяцев `legacy` архивируется

Почему: V2 по факту — свалка. Попытка её аккуратно разобрать автоматически — недели работы с сомнительным выходом. Лучше честно признать «старое — legacy, новое — правильное» и двигаться.

## NG-13. Qoopia не делает собственную авто-саммаризацию как primary path

**Добавлено в Фазе 1.5 (2026-04-11)**.

Что это значит:
- Qoopia **не** запускает фоновый воркер который сама зовёт Haiku/GPT чтобы саммаризировать сессии
- Qoopia **не** имеет конфигурации summarization model/provider
- Qoopia **не** держит auth cascade для отдельного summarization-LLM
- Qoopia **не** реализует three-level escalation (normal → aggressive → truncate)

Почему: это огромная подсистема (auth profiles, provider fallback, retry, error handling, cost tracking для summarization), которая **не нужна** если агент сам вызывает `session_summarize()` со своим готовым текстом. Агент и так живой в этот момент, и так платит за свою модель, и так знает что важно — пусть сам напишет саммари.

Что это НЕ запрещает:
- Агент **может** сам саммаризировать свою историю — это основной путь, через `session_summarize(content=<мой текст>)`
- В V3.5+ может появиться **опциональный** background worker как opt-in per-workspace, если окажется что агенты систематически забывают суммаризировать
- По умолчанию в V3.0 его не будет никогда

## NG-14. В V3.0 нет semantic-поиска, нет embeddings, нет vector storage

**Добавлено в Фазе 1.5 (2026-04-11)**. Это самое радикальное упрощение всего Simplicity Pass.

Что это значит:
- V3.0 **не** использует embedding-модели (ни OpenAI API, ни локальные)
- V3.0 **не** имеет vector store (ни sqlite-vec, ни pgvector, ни Qdrant)
- V3.0 **не** делает semantic similarity search
- V3.0 **не** делает RAG в смысле «векторные эмбеддинги + cosine similarity»
- V3.0 **весь retrieval через FTS5** (SQLite built-in full-text search)

Почему: корректировка масштаба knowledge base (1 ГБ реальный, 10 ГБ как ceiling) показала что **FTS5 достаточно** для типичных объёмов. Embedding-инфраструктура добавляет огромную сложность (провайдер, auth, cost, chunking pipeline, re-indexing при обновлениях), которая оправдана только когда FTS5 **реально не хватает**. В V3.0 мы это ещё не доказали — значит не строим.

Что это НЕ запрещает:
- V3.5+ может добавить Layer B (semantic) как **отдельный модуль** над существующим Layer A (FTS5). Это аддитивное изменение.
- Триггером для Layer B станет **реальная боль** реального пользователя: «FTS5 не находит по смыслу, приходится пробовать 10 вариантов запроса». Не «а было бы круто иметь».
- Архитектура V3.0 **спроектирована так**, чтобы Layer B можно было добавить без переписывания Layer A — это отдельный ADR в Фазе 3.

Практическое следствие для пользователей: `recall("как вернуть товар")` в V3.0 не найдёт абзац «процедура возврата» — нужно искать по ключевым словам. Агент должен это знать и перебирать варианты.

## NG-15. В V3.0 нет large file interception

**Добавлено в Фазе 1.5 (2026-04-11)**.

Что это значит:
- V3.0 **не** перехватывает файлы > N токенов в message content
- V3.0 **не** хранит отдельные большие файлы в выделенной папке `~/.qoopia/large-files/`
- V3.0 **не** генерирует exploration summaries для файлов
- V3.0 **не** имеет `session_describe(file_id)` tool

Вместо этого — **жёсткое правило**: `session_save` принимает content ≤ 100 KB. Больше — возвращает ошибку с инструкцией «разбей на части или храни файл снаружи, передай в Qoopia только ссылку/путь».

Почему: это целая подсистема (отдельная таблица, storage layer, интерcept при ingestion, exploration-summaries pipeline), которая **в V3.0 не нужна** — агент может сам хранить большие файлы в файловой системе tenant'а и класть в Qoopia только ссылки + выжимки.

Что это НЕ запрещает:
- V3.5+ может добавить file storage слой если реально начнутся жалобы
- Knowledge base ingestion для больших мануалов — отдельная задача в V3.5, завязана на Layer B

## NG-12. Фичи мониторинга/observability — минимум, не максимум

Что это значит:
- **Нужно**: базовые метрики (latency percentiles, uptime, ошибки, token savings из C1)
- **Нужно**: простой health check
- **Не нужно**: полноценный APM, distributed tracing, Grafana dashboards, alerting rules на каждый метрик
- **Не нужно**: отдельная observability-подсистема как продукт

Почему: Qoopia сама по себе — не гигант. Её мониторинг должен быть соразмерен. Overengineering убьёт простоту деплоя (NG-9 и Группа D в `04-success-criteria.md`).

## Пересечение с другими системами (чего там быть не должно)

Эта таблица помогает сказать «стоп, это не наш скоуп» при сомнениях.

| Зона | Кто занимается | Qoopia |
|---|---|---|
| Email content | Gmail / другой email MCP | Только task-bound выжимки |
| Calendar events | Calendar MCP | Только task-bound выжимки |
| Code editing | IDE + file tools | Не касается |
| Deployment / DevOps | launchd / systemd / CI | Qoopia только *использует*, не управляет |
| Authentication | Внешний IdP | Не касается, только internal workspace ACL |
| Analytics | Отдельная BI, если будет | Не касается |
| Background jobs | Внешний job runner, если будет | Не касается |
| Real-time chat | Telegram / Slack / WhatsApp | Не касается |

## Как использовать этот документ

При любом предложении «а давайте добавим X» — сначала открываем этот файл. Если X попадает в любой из NG-N — не добавляем. Если X **может быть** расценено как NG, но есть сомнения — обсуждаем явно и либо фиксируем исключение (ADR), либо отказываемся.

Это защита от «расползания скоупа».
