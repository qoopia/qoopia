# Фаза 1.5 — Simplicity Pass

**Триггер**: изучение lcm-mcp товарища Асхата (2026-04-11) показало, что наш набросок Qoopia V3 содержит over-engineering. Нияз сделал рабочую замену LCM за 665 строк и 2 файла; наш мысленный набросок тянет на порядки больше. Если скрытая сложность есть в одном месте — она гарантированно есть и в других.

**Цель фазы**: пересмотреть всё что написано в 01-04 через призму «радикальной простоты» и вырезать всё, что не является абсолютно необходимым для primary acceptance test (коллапс system prompt агента в одну строчку «проверь Qoopia» с железобетонным доверием).

**Принцип фазы**:
> Любое решение должно быть **самым простым** из всех альтернатив, которые удовлетворяют принципам. Сложность требует явного обоснования, простота — нет.

**Фикс по масштабу knowledge base**: 1 ГБ реальных данных, 10 ГБ был запас «на будущее если Qoopia станет массовым RAG-решением». Это меняет расчёт по Layer A / Layer B: FTS5 в SQLite на 1 ГБ — зона комфорта, а не край возможностей.

## Scope фазы — что именно пересматриваем

### 1. Слой retrieval — ✅ уже решено

- **V3.0 = только Layer A (FTS5) + entities + workspaces**
- **Layer B (semantic) откладывается в V3.5** по факту боли
- Это уже подтверждено владельцем 2026-04-11

Остаётся только отразить это в документах 01-04.

### 2. Workspace model — пересмотр нужен

**Сейчас заложено**: 3 режима подключения агента (shared / scoped / autonomous).

**Вопрос для пересмотра**: нужны ли все три в V3.0, или хватает двух, или даже одного?

**Гипотеза Simplicity Pass**: в текущем парке Асхата **ни один агент не использует shared и scoped**. Каждый агент живёт в своём собственном workspace (Alan → personal, Aizek → kz-ops, Aidan → america-ops, Dan → family, Claude → claude-sessions). Это всё **autonomous**. Доступ к knowledge-base — это **не scoped**, это отдельное измерение (глобальный read-only ресурс для всех). Cross-workspace read для Claude — это **не scoped**, это privilege агента.

Если так — то V3.0 имеет **один режим** (autonomous workspace per agent) + **два ортогональных механизма** (knowledge-base flat global + Claude cross-read privilege). Это радикально проще.

**Вопрос**: shared mode в V3.5 появится по реальной боли (несколько агентов одновременно пишут в один workspace), или вообще не понадобится?

### 3. Entity model — пересмотр нужен

**Сейчас заложено**: tasks, deals, contacts, finances, projects — отдельные таблицы, перенесены из V2.

**Вопрос для пересмотра**: нужны ли отдельные таблицы для каждого типа сущности, или достаточно **одной таблицы `notes` с тегом типа и структурированными метаданными** в JSON?

**Аргумент за упрощение**: агенту всё равно — он пишет и читает через одни и те же MCP tools. CRUD по «task» и по «deal» отличается только набором полей. Одна таблица `notes` + JSON metadata + индексы по ключевым полям = то же самое снаружи, в 5 раз меньше SQL.

**Аргумент против упрощения**: структурированные таблицы позволяют быстрые range-запросы (все задачи со сроком в июне), которые JSON-метаданные делают медленнее.

**Кто из агентов сейчас реально делает range-queries по entities?** Ответ нужен фактический — если никто, упрощаем. Если кто-то — оставляем структурированно.

### 4. Retention policy — пересмотр нужен

**Сейчас заложено**: LRU + importance scoring + summarization before delete + manual pins + task-bound auto-purge (5 разных механизмов).

**Гипотеза Simplicity Pass**: в V3.0 хватает **двух правил**:
1. Task-bound контекст стирается при закрытии задачи (из 02-personas.md)
2. Всё остальное живёт пока явно не удалили

LRU, importance scoring, summarization-before-delete — это V4+ по факту боли. Manual pins — тоже V4+, пока «всё что записано, живёт» это и есть pinning.

**Вопрос**: можем принять такое упрощение?

### 5. Large file handling — пересмотр нужен

**Сейчас заложено**: перехват больших файлов (>25K токенов), выделение в отдельное хранилище, compact reference в сообщение.

**Гипотеза Simplicity Pass**: в V3.0 — **не делаем** эту подсистему. Правило: «не пиши файл > 25K токенов через `session_save` — агент сам разбивает или хранит файл снаружи и кладёт ссылку». Это выкидывает целую подсистему (large_files таблица, папка хранилища, exploration summaries, `session_describe` по file_id).

Если агент реально упрётся — добавляем в V3.5.

**Вопрос**: принимаем?

### 6. Auto-compaction — пересмотр нужен

**Сейчас заложено**: Qoopia в фоне сама компактирует старые чанки через Haiku (fallback для случаев когда агент не вызвал `session_summarize` сам).

**Гипотеза Simplicity Pass**: в V3.0 — **не делаем**. Следуем пути Нияза: агент сам пишет саммари, сам вызывает `session_summarize`. Если забывает — в system prompt напоминание. Если всё равно забывает — это проблема инструкции, не системы.

Убираем: background worker, Haiku auth cascade, three-level escalation, provider config. Примерно 30% потенциальной сложности Qoopia улетает.

**Вопрос**: принимаем?

### 7. Multi-tenant isolation — пересмотр реализации

**Сейчас декларация**: workspace = граница изоляции.

**Вопрос**: реализация — самый тупой вариант, **один столбец `workspace_id` + `WHERE` в каждом запросе**. Без row-level security, без отдельных SQLite-баз на workspace, без prefix-based routing. Один столбец, один WHERE, всё.

**Когда усложнять**: если появится реальный сценарий «два абсолютно несвязанных tenant'а на одной инсталляции с гарантиями юридического уровня». Для Асхат + Сауле + гипотетические early adopters — простой столбец покрывает.

### 8. Install & deployment — пересмотр

**Сейчас заявлено**: `qoopia install` ≤ 5 минут.

**Бюджет Simplicity Pass**:
- **≤ 3 команды** до работающего MCP сервера
- **≤ 2 минуты** на чистой машине
- **0 конфигов** в default установке
- **≤ 5 runtime зависимостей**
- Runtime: Bun (как у Нияза) или Node.js — решаем в Фазе 3, но первое — кандидат по умолчанию, потому что SQLite встроен
