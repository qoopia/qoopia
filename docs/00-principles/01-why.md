# 01 — Зачем Qoopia существует

**Status**: ✅ v2 signed off by Асхат 2026-04-11
**Источник**: прямые ответы Асхата в Phase 1 dialog

## В одном предложении

> **Qoopia — это token-economy layer для агентов: внешний склад памяти и знаний, позволяющий LLM оставаться максимально худыми и дешёвыми.**

## Три функции, ранжированные по важности

### 1. Token economy (главное)

Агент **не должен грузить всю историю в system prompt**. Вместо этого:

- **На старте сессии**: агент за копейки вызывает `recall()` точечно — только то что нужно для текущей задачи. Нет "на всякий случай" контекста.
- **По ходу сессии**: когда контекст переполняется, агент сбрасывает промежуточные результаты в Qoopia через `note()` и продолжает работу с чистой головой.
- **Measured savings**: каждый такой вызов должен давать видимый выигрыш в токенах по сравнению с "грузить всё в prompt".

Это **primary function**. Всё остальное служит ей.

### 2. Память между чатами

Когда Асхат запускает новую сессию — агент должен **помнить предыдущее общение** без того чтобы Асхат пересказывал это руками. Это следствие (1): сохранение состояния между сессиями — частный случай сбрасывания контекста.

Самая больная потеря если Qoopia исчезнет: **каждая сессия начинается с нуля, токены сгорают на повторные объяснения одного и того же**.

### 3. Knowledge base (справочник большого объёма)

Qoopia хранит **большие документы**, которые агент не может держать в контексте:
- История прошлых сессий
- Тысячестраничные мануалы
- Руководства, глоссарии
- Регламенты, процедуры, инструкции

Агент **никогда не грузит их целиком**. Вместо этого делает precise retrieval ("найди параграф про возврат товара в мануале Wildberries") и получает **только релевантный кусок** — обычно 1-2 абзаца.

## Design stance: боль своя, решение общее

Qoopia V3 **проектируется чтобы закрыть конкретные боли Асхата** — всё перечисленное выше (token economy, память между сессиями, RAG по большим мануалам). Это **источник требований**, а не ограничение архитектуры.

Но **реализация делается так, чтобы любой, кто захочет взять это решение, мог его развернуть и пользоваться**:
- Эффективно — latency, cost, качество retrieval не хуже чем у нас
- Безопасно — multi-tenancy, изоляция данных между пользователями/organizations, никакой утечки из чужого workspace
- Стабильно — понятная миграция, бэкапы, rollback, observability

**Практическое следствие**: никаких хардкодов "askhat", никаких путей `/Users/askhatsoltanov/`, никаких "у меня один пользователь значит можно". С первой строчки кода — multi-tenant ready, даже если сейчас tenant один.

## Главный пользователь

**Агенты** (Alan, Aizek, Aidan, Claude, будущие Zoe, Mia, Dan). Не человек. Не сторонние системы.

Именно LLM-агенты читают и пишут в Qoopia во время работы. Человек (Асхат) и сторонние системы — вторичные consumers.

## Три технических следствия

Эти три пункта — не решения, а **критерии для всех дальнейших решений**. Любая архитектура, которая им противоречит, отбрасывается.

### Следствие 1: Latency критична

Если `recall()` медленный — агент не будет им пользоваться. Будет грузить "на всякий случай" в prompt. Это убивает token-economy функцию.

**Бюджет**: recall должен укладываться в сотни миллисекунд, не секунды. Это ограничивает выбор embedding-модели, backend, индексов.

### Следствие 2: Chunking для больших документов

Тысячестраничный мануал не хранится как одна запись. Нужна стратегия:
- Разбиение на смысловые куски (параграфы? секции? скользящее окно?)
- Embedding каждого куска отдельно
- Retrieval возвращает **нужный кусок**, а не весь документ
- Опционально: возврат соседних кусков для контекста

Это отдельная архитектурная задача. Фиксируется в TO-BE.

### Следствие 3: Cost visibility

Агент должен видеть "сколько токенов я только что сэкономил этим recall". Иначе невозможно понять, работает ли экономия. Нужен **явный учёт**:
- Размер извлечённого куска (tokens)
- Что было бы если грузить всё целиком (tokens)
- Экономия (%)

Это нужно и для мониторинга системы, и для самого агента ("стоит ли пользоваться или проще заинжектить в prompt").

## Что это меняет по сравнению с текущей V2

Текущая V2 — **операционная CRM с notes-в-довесок**. У неё:
- 46 таблиц, FTS5 индексы на всех 6 сущностях с равным весом
- MCP tools одинаково обслуживают tasks/deals/contacts/notes
- `recall()` обрезает текст до 300 символов — **противоречит token-economy функции напрямую** (если можно получить только 300 символов, невозможно вернуть полный параграф из мануала)
- Нет chunking для больших документов
- Нет учёта cost per query
- Нет различия "маленькая note агента" vs "кусок из большого мануала"

**V3 переворачивает приоритеты**:
- **Notes + knowledge base = first class** (специально оптимизированы для token-economy)
- **Tasks/deals/contacts = reference entities** (нужны, но не определяют архитектуру)
- **Recall возвращает столько, сколько нужно** (никаких substring(0, 300))
- **Chunking — встроенная часть pipeline** для больших документов
- **Cost metrics видны** в каждом ответе

## Что это НЕ меняет

- Notes могут ссылаться на tasks/deals/contacts/projects — это связки, не подчинение
- Tasks/deals/contacts/finances всё ещё нужны, но **не определяют архитектуру**
- Dashboard для человека может существовать, но **проектируется после того как агентская часть работает** (сейчас dashboard реализован и бесполезен — подтверждено владельцем)

## Scope ownership (кто что владеет)

| Зона | Владелец | Статус в V3 |
|---|---|---|
| Session memory (между чатами) | Primary | 🔥 core |
| Knowledge base (мануалы, глоссарии) | Primary | 🔥 core |
| Operational entities (tasks, deals, contacts) | Secondary | ✅ supported |
| Dashboard для человека | Tertiary | ⚠️ redesign нужен, но позже |
| REST API для сторонних систем | Tertiary | ✅ supported |

## Открытые вопросы, уже отвеченные

| Вопрос | Ответ Асхата | Фиксация |
|---|---|---|
| Общая vs изолированная память | "Пусть видят всё, но не понимаю рисков" | **Гибрид**: write=own, read=own by default, explicit cross-agent opt-in, knowledge base всегда shared. ADR в Фазе 3. |
| Retention policy | "Точно нужна, но нужен грамотный механизм" | **Да**. Отдельный ADR в Фазе 3: LRU + importance scoring + summarization before delete + manual pins. |
| Recall precision vs browse | "И то и то важно" | **Оба**: точный semantic recall + browse-by-filter (time/agent/project/type). В API два разных endpoint. |
| Human dashboard | "Реализован, бесполезен, сам не знаю как надо" | Redesign в Фазе 3 **после** того как агентская часть готова. UX-воркшоп отдельно. |
| Non-LLM consumers | Дашборд нужен, REST вторично | REST сохраняется как supported but not driving, dashboard — отдельный deliverable. |

## Ответы владельца (2026-04-11)

- **Масштаб knowledge base**: до **10 ГБ**. Это серьёзно, это не "положить в sqlite и забыть" — это архитектурное ограничение для Фазы 3.
- **Latency**: "максимально быстро". Фиксируем как принцип; конкретные цифры SLA — в `04-success-criteria.md`.
- **Multi-tenant from day one**: см. раздел "Design stance" выше.

## Остаются на Фазу 3 (архитектура)

Эти вопросы не про "зачем", они про "как". Обсудим когда дойдём до TO-BE:

- Embedding model (OpenAI API vs локальная) — вопрос денег и латентности
- Vector storage (sqlite-vec vs pgvector vs отдельный vector DB) — вопрос масштаба при 10 ГБ
- Кто инициирует первую запись в сессии (агент/автоматика/человек) — вопрос UX первого контакта
