# 04 — Что значит «Qoopia работает хорошо»

**Status**: ✅ v1 signed off by Асхат 2026-04-11
**Источник**: производные от `01-why.md` (token economy), `02-personas.md` (multi-tenant, workspace model), `03-use-cases.md` (6 UC + 7 требований железобетонности), прямые цитаты Асхата.

Документ переводит общие принципы в **числа и проверяемые критерии**. Если Qoopia V3 не попадает хотя бы в один критерий — она ещё не готова. Если попадает во все — primary acceptance test пройден и можно заменить system prompt любого агента на «проверь Qoopia».

Важная оговорка: **все числа ниже — мои предложения**. Они обоснованы из того что я знаю про LLM-агентов, embedding-модели и пользовательское восприятие. Там где цифра спорная, я это явно отмечаю. Владелец может ужесточить любую или ослабить любую — но явно, с записью в документ.

## Группа A — Надёжность (железобетон)

Самая важная группа. Без неё остальное не имеет значения.

| # | Критерий | Порог | Почему |
|---|---|---|---|
| A1 | **Запись не теряется** | 0 потерь при штатной работе. Ни одного случая «я сохранил, а потом не нашлось». | `note()` вернул успех → заметка существует. Без «но». |
| A2 | **Recall находит то что есть** | ≥ 99.5% запросов по семантике возвращают релевантный результат, если он в базе | Если recall иногда «не видит» свою же запись — доверие рушится, агент перестаёт пользоваться, token economy умирает. |
| A3 | **Никакой truncation посередине** | 0 случаев обрезки текста по произвольному лимиту | Это прямая причина сегодняшней боли с передачей контекста между чатами Claude. **Хардкод-лимитов на длину возвращаемого поля в V3 не существует.** Пагинация — да, обрезка — никогда. |
| A4 | **Uptime MCP-сервера** | ≥ 99.9% за месяц (≈ 43 минуты простоя в месяц максимум) | Агенты обращаются к Qoopia десятки/сотни раз в день. Каждая минута простоя — агенты не могут работать. |
| A5 | **Целостность данных при сбое** | 0 случаев частично записанных/повреждённых заметок после crash/kill/питание | Транзакционность write-операций. Atomic writes. Если процесс упал в середине `note()` — либо заметка есть целиком, либо её нет совсем. Третьего не существует. |
| A6 | **Бэкапы** | Автоматический бэкап ≤ 1 раз в сутки. Восстановление из любого бэкапа за ≤ 10 минут ручной работы. | «Железобетонно» = восстановление после любого сценария, включая «дропнули БД». |
| A7 | **Миграции схемы не ломают данные** | 100% миграций применяются автоматически без ручного вмешательства. Rollback возможен на любую предыдущую версию за ≤ 5 минут. | Обновление Qoopia не должно стоить суток работы и страха. |

## Группа B — Latency

Из `01-why.md`: «recall должен укладываться в сотни миллисекунд, не секунды». Из round 2: «максимально быстро». Перевожу в числа.

| # | Операция | p50 (типичная) | p95 (почти всегда) | p99 (худший допустимый) |
|---|---|---|---|---|
| B1 | `recall()` semantic, база ≤ 100 МБ текста | ≤ 100 мс | ≤ 250 мс | ≤ 500 мс |
| B2 | `recall()` semantic, база 1–10 ГБ текста | ≤ 300 мс | ≤ 600 мс | ≤ 1000 мс |
| B3 | `note()` запись до 10 КБ | ≤ 50 мс | ≤ 150 мс | ≤ 300 мс |
| B4 | `note()` запись 10–100 КБ | ≤ 150 мс | ≤ 400 мс | ≤ 800 мс |
| B5 | `list()` / `get()` entity по id | ≤ 30 мс | ≤ 100 мс | ≤ 200 мс |
| B6 | `brief()` full workspace context | ≤ 500 мс | ≤ 1000 мс | ≤ 2000 мс |

**Обоснование чисел**: агент делает 5–20 обращений в одной логической задаче. Если каждое стоит 1 секунду, задача стоит 20 секунд задержки — ощутимо для человека, неприемлемо для агентов. При сотнях миллисекунд — задержка суммарно под секунду, незаметна.

**Ограничение**: эти числа включают сетевой round-trip MCP. То есть от момента когда агент сформировал запрос до момента получения ответа. Не только «сам движок».

**Спорная цифра**: B2 p99 1000 мс. Возможно стоит ужесточить до 700 мс. Зависит от выбора vector store (Фаза 3). Сейчас фиксирую как «заявка»; проверим выполнимость при проектировании архитектуры.

## Группа C — Token economy (экономия)

Это primary function из `01-why.md`. Должна быть **видимой и измеримой**, иначе непонятно работает ли.

| # | Критерий | Порог | Как измеряем |
|---|---|---|---|
| C1 | **Cost visibility в каждом ответе recall** | Каждый ответ содержит: размер найденного фрагмента (токены), оценка «сколько было бы если грузить всё» (токены), экономия (%) | Иначе невозможно понять работает ли token economy. Из `01-why.md` «Следствие 3». |
| C2 | **Реальная экономия на типичной задаче** | ≥ 70% токенов по сравнению с «грузить всё в prompt» для задач где применимо | Это целевая эффективность. Если меньше — агент всё равно будет инжектить в prompt «на всякий случай». |
| C3 | **Минимальный system prompt агента** | Стандартный system prompt агента в V3 ≤ 500 токенов (персона + 1 строчка «всё остальное в Qoopia») | Это **числовая версия** мечты владельца про «system file в одну строчку». 500 токенов — базовая персона и правила, не контекст. Весь контекст приходит из Qoopia. |

## Группа D — Простота развёртывания (multi-tenant proof)

Из `02-personas.md`: единственный gate для развёртывания Qoopia у Сауле — работающий one-command deploy. Это же — proof «годности для всех».

| # | Критерий | Порог | Почему |
|---|---|---|---|
| D1 | **Одна команда установки** | `qoopia install` (или эквивалент) с нуля до работающего MCP-сервера ≤ 5 минут | Если сложнее — не multi-tenant, а «собрали для Асхата». |
| D2 | **Zero config files для базового случая** | Первый запуск работает без редактирования ни одного конфига. Кастомизация — опциональна. | Новый пользователь не должен читать доку перед первым успехом. |
| D3 | **Идентичность деплоев** | У Асхата, Сауле и любого третьего человека после `qoopia install` — идентичная версия, идентичное поведение | Никаких «у меня работает, у тебя что-то не то». |
| D4 | **Подключение нового агента** | Одна команда `qoopia attach-agent <name> --workspace <mode>` без правок чужих файлов | Из `02-personas.md`: «хоть 15 агентов, хоть удалили 20». |
| D5 | **Отключение агента** | Одна команда `qoopia detach-agent <name>`. Опция сохранить или удалить home workspace. | То же. |

## Группа E — Retrieval quality (качество семантики)

Это критичная, но самая сложная для измерения группа. Без неё система быстро найдёт **не то**.

| # | Критерий | Порог | Как измеряем |
|---|---|---|---|
| E1 | **Recall@5 на золотом наборе** | ≥ 90% запросов на golden set возвращают правильный результат в топ-5 | Золотой набор — 50-100 ручных вопросов-ответов которые владелец пометит «вот это должно находиться». Создаётся в Фазе 2. |
| E2 | **Precision на типичном запросе** | Среди топ-5 результатов ≥ 3 релевантные | Мусора быть не должно. |
| E3 | **Chunking больших документов работает** | Мануал в 500 страниц разбивается и retrieval возвращает правильный чанк для 90% запросов по мануалу | Из `01-why.md` «Следствие 2». Фиксируем числовую версию. |
| E4 | **Семантика реально семантическая** | Запрос «как вернуть товар» находит абзац «процедура возврата» (синонимы) | Не только full-text match. Иначе нет смысла городить embeddings. |

## Группа F — Безопасность и изоляция

Из `02-personas.md`: workspace = граница изоляции. Надо проверять её автоматически.

| # | Критерий | Порог | Почему |
|---|---|---|---|
| F1 | **Cross-workspace read без явной подписки невозможен** | 0 случаев когда агент workspace X читает запись из workspace Y без явной привилегии | Основное обещание tenancy model. Тест обязателен. |
| F2 | **Cross-workspace write невозможен** | 0 случаев записи агента в чужой home | Из решения по Claude: даже privilege «cross-workspace reader» не даёт права писать. |
| F3 | **Email task-bound автоочистка работает** | 100% записей привязанных к закрытой задаче удаляются в течение 1 часа после закрытия | Из `02-personas.md` и 03-use-cases.md UC-4. |
| F4 | **Нет tenant-leak через логи** | В логах/метриках невозможно увидеть содержимое чужого workspace | Логи могут содержать workspace_id, но не содержимое. |

## Группа G — Качество жизни (UX для агента)

Эти критерии — не про жизнь или смерть, но если они плохи, агент всё равно будет искать способ обойти Qoopia.

| # | Критерий | Порог | Почему |
|---|---|---|---|
| G1 | **Понятные ошибки** | Любая ошибка содержит: что хотел, что пошло не так, что делать дальше. Ни одного «internal server error» без контекста. | LLM-агент не может отладить непонятную ошибку, он просто не использует tool дальше. |
| G2 | **Tools MCP ergonomic для LLM** | ≤ 10 tools на типичный workflow. Понятные имена. Нет перегрузок типа «list() с 15 флагами». | В V2 было 29 → консолидировали до 8 (см. старую работу). V3 — сохраняем этот уровень. |
| G3 | **Дефолты разумные** | Каждый параметр с дефолтом, который работает в 80% случаев. Явно указывать — только когда нужно отступить от дефолта. | Агент не должен читать доку каждый раз. |

## Как проверяем всё это

1. **Automated test suite** — критерии A1-A7, B1-B6, D1-D5, F1-F4 автоматизируются и прогоняются в CI при каждом изменении.
2. **Golden set + eval runner** — для E1-E4 делаем ручной золотой набор и прогон метрик качества.
3. **Живой мониторинг** — A4 (uptime), B (latency percentiles), C1 (visibility в ответах) — метрики в продакшне.
4. **Acceptance test у Сауле** — D1-D5 проверяются вживую когда Qoopia деплоится ей.
5. **Primary acceptance test**: Асхат берёт одного из своих агентов, удаляет из его system prompt всё кроме «проверь Qoopia», запускает работу. Если всё работает как раньше — V3 принята.

## Чего тут сознательно нет

- **Throughput** (запросов в секунду). Пока это не боль. Когда станет — добавим.
- **Качество embeddings** как отдельная группа. Поглощено E1-E4 (smoke через retrieval quality).
- **Developer experience** для тех кто пишет код Qoopia. Важно, но не principle, это уже Фаза 3.
