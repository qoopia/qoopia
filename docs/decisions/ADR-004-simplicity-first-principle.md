# ADR-004: Radical simplicity as first-class cross-cutting principle

**Status**: accepted
**Date**: 2026-04-11
**Deciders**: Асхат + Claude

## Контекст

Фаза 1 Principles Qoopia V3 изначально не имела явного принципа «простота». Железобетонная надёжность была декларирована (`03-use-cases.md` — 7 требований), но simplicity была неявным значением.

Изучение lcm-mcp Нияза Ирсалиева 2026-04-11 показало что наш первоначальный набросок Qoopia V3 нёс несоразмерную сложность (over-engineering) в нескольких подсистемах: workspace modes, entity model, retention policy, auto-compaction, large file handling, multi-tenant isolation.

Если скрытая сложность есть в одном месте — она гарантированно есть и в других, если нет явного принципа её отсекать.

## Варианты

### Вариант A (самый простой возможный): явный принцип + бюджеты + обязательная проверка в ADR

- Добавить принцип «радикальная простота» равноправно с «железобетонность» в `01-why.md`
- Добавить числовые бюджеты (LoC, deps, tables, commands) в `04-success-criteria.md` как группу H
- Обновить ADR template чтобы требовать минимум 3 варианта включая «самый простой возможный»
- Каждый новый ADR обязан пройти проверку «выбран самый простой вариант или объяснено почему нет»

- Плюсы:
  - Явно, проверяемо, невозможно «забыть»
  - Встроено в процесс принятия решений
  - Числовые бюджеты → объективно, не зависят от вкуса
- Минусы:
  - Требует дисциплины при написании ADR
  - Может иногда конфликтовать с надёжностью — нужен arbitration

### Вариант B: неявный принцип, полагаемся на здравый смысл

- Плюсы: нет формальностей
- Минусы: именно это уже случилось и привело к over-engineering. Не работает.

### Вариант C: simplicity как soft рекомендация в одной строке принципов

- Плюсы: символическая declaration без формализма
- Минусы: ничего не меняет по сравнению с Вариантом B

## Решение

Выбран **Вариант A**.

**Формулировка принципа** (фиксируется в `01-why.md`):

> Любое решение в Qoopia V3 должно быть **самым простым** из всех альтернатив, которые удовлетворяют принципам. Если существует более простая альтернатива, которая тоже удовлетворяет — мы выбираем её. Сложность требует явного обоснования, простота — нет.

**Operationalization**:

1. **Группа H «Simplicity budgets»** в `04-success-criteria.md` с числовыми целями (LoC, deps, tables, commands, install time, system prompt size, tool count)

2. **ADR template обновлён** (`ADR-000-template.md`): обязательна секция с минимум 3 вариантами, один из которых «самый простой возможный». Отклонение от самого простого требует явного объяснения.

3. **Arbitration при конфликте надёжности и простоты**: надёжность выигрывает **только** если конкретно обосновано что более простой вариант её нарушает. По умолчанию — простота.

**Проверка на простоту**: Вариант A — самый простой из тех что реально работают. Вариант B — ещё проще, но уже доказано что не работает (именно эта проблема нас и привела сюда). Вариант A выбран.

## Последствия

### Что становится проще

- Будущие ADR будут систематически ловить over-engineering на этапе написания, а не после реализации
- Числовые бюджеты дают объективный критерий «это слишком сложно»
- Новый агент / новый разработчик, работающий с проектом, видит явное правило и следует ему

### Что становится сложнее

- Писать ADR — чуть дольше, потому что нужно перечислить и оценить минимум 3 варианта
- Иногда приходится отстаивать «это действительно самый простой из возможных», а не «мне так привычно»
- Возможны tension'ы между железобетонностью и простотой — каждая такая ситуация требует явного решения и ADR-записи

### Что мы теперь не сможем сделать

- «Просто добавить фичу потому что она кажется полезной» — теперь надо показать что это самый простой способ решить конкретную задачу, и что альтернативы отвергнуты
- Копировать паттерны из сложных систем «потому что они стандарт» без обоснования

### Что нужно будет пересмотреть

- Если через 6 месяцев окажется что simplicity-first приводит к постоянным пересборкам и refactor'ам из-за слишком агрессивного YAGNI — ослабим формулировку. Сейчас исходим из того что YAGNI агрессивный — правильная настройка.
