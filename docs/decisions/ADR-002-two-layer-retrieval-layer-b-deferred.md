# ADR-002: Two-layer retrieval architecture, Layer B deferred to V3.5

**Status**: accepted
**Date**: 2026-04-11
**Deciders**: Асхат + Claude

## Контекст

Первоначальный набросок Qoopia V3 предполагал единый слой retrieval на основе embedding-поиска (semantic similarity), с `recall()` как основным инструментом. Это тянуло за собой:

- Выбор embedding-провайдера (OpenAI API vs Voyage vs локальная модель)
- Выбор vector store (sqlite-vec vs pgvector vs Qdrant/Weaviate)
- Auth cascade для embedding API
- Cost management (каждый embed стоит денег)
- Chunking pipeline для больших документов
- Re-indexing при обновлении записей

Изучение peer implementation (lcm-mcp от Нияза Ирсалиева, 2026-04-11) показало что для session memory и небольших knowledge bases **FTS5 в SQLite достаточно** и радикально проще. Его реализация работает, полезна пользователям и не использует embeddings вообще.

Дополнительный фактор: владелец уточнил что реальный масштаб knowledge base — **1 ГБ**, а не 10 ГБ (10 было с запасом). 1 ГБ текста в FTS5 — зона комфорта.

## Варианты

### Вариант A (самый простой возможный): только FTS5, никакого semantic

- Плюсы:
  - Ноль инфраструктурной сложности (SQLite built-in)
  - Ноль cost-per-query
  - Sub-100 мс latency на 1 ГБ базе
  - Install = 2 команды
  - Отсутствие провайдер-зависимости (не ломаемся когда OpenAI лежит)
- Минусы:
  - Не находит по синонимам («как вернуть товар» не найдёт «процедура возврата»)
  - Требует от агента пробовать несколько keyword-вариантов
  - Слабее для русского языка с его морфологией (частично компенсируется prefix-matching)
- Почему отвергнут (если отвергнут): **не отвергнут, выбран для V3.0**

### Вариант B: гибрид FTS5 + semantic с самого начала

- Плюсы:
  - Лучшее качество retrieval на первом релизе
  - Покрывает и keyword и synonymy
  - Profesionально выглядит
- Минусы:
  - Удваивает инфраструктурную сложность
  - Требует решения по embedding-провайдеру **до** того как знаем реальные объёмы
  - Install становится сложнее (нужен API key / локальная модель)
  - Cost-per-query ненулевой
  - Ставит под угрозу simplicity budget (H1-H5)

### Вариант C: только semantic, без FTS5

- Плюсы:
  - Один код-path для retrieval
  - Максимальное качество «по смыслу»
- Минусы:
  - Вся инфраструктурная сложность embedding-подхода
  - Медленнее чем FTS5 для простых keyword-запросов
  - Fragility — провайдер лежит = retrieval не работает

## Решение

Выбран **Вариант A** для V3.0.

**Two-layer архитектура** как принцип:
- **Layer A (Session memory + Notes + Entities)**: SQLite + FTS5, никаких embeddings. Это ядро V3.0.
- **Layer B (Semantic knowledge base)**: откладывается в V3.5. Спроектирован как **аддитивный** слой над Layer A — не требует переписывания Layer A при добавлении.

**Триггер для работы над Layer B**: реальная боль реального пользователя. Формулировка: «я пробовал 5 keyword-вариантов, ничего не нашёл, хотя знаю что информация в базе есть». Когда такие жалобы пойдут — начинаем Layer B в V3.5.

**Проверка на простоту**: Вариант A — самый простой. Удовлетворяет принципам (token economy, железобетонность, простота). Выбран без компромиссов.

## Последствия

### Что становится проще

- Install: `bun install && bun run src/index.ts` (или эквивалент) — 2 команды, 0 конфигов
- Deployment: zero external dependencies, один SQLite-файл содержит всё
- Latency: p99 в диапазоне 50-400 мс реально достижимы (vs заявленные 1000 мс для semantic)
- Cost: $0 per query
- Install budget H5 (≤ 3 команды) реалистичен
- Setup на стороне Сауле (gate для Фазы 5) реально простой

### Что становится сложнее

- Агент должен знать что `recall` — keyword-based, и уметь перебирать варианты запроса
- Русский язык с его морфологией требует prefix-matching для нормальной работы
- Поиск «по смыслу» в V3.0 недоступен — это осознанный trade-off

### Что мы теперь не сможем сделать

- Cross-lingual retrieval (запрос на русском на английский документ)
- Fuzzy semantic matching без keyword совпадения
- RAG в традиционном смысле с вектор-поиском

### Что нужно будет пересмотреть если изменятся условия

- Если у пользователя knowledge base > 10 ГБ и FTS5 начинает тормозить — это триггер для работы над Layer B раньше V3.5
- Если появится реальная боль от отсутствия synonym-матчинга — Layer B
- Если встроенный SQLite FTS5 окажется неподходящим по какой-то технической причине — пересматриваем бэкенд Layer A (но не добавляем Layer B без необходимости)
