# ADR-001: Отдельный workspace `~/qoopia-v3/`, прод-Qoopia не трогаем

**Status**: accepted
**Date**: 2026-04-11
**Deciders**: Асхат + Claude

## Контекст

Прод-Qoopia (`~/.openclaw/qoopia/`) live. На ней завязаны все агенты (Aidan, Aizek, Alan), cron-джобы и MCP клиенты. Любое изменение без плана = риск положить агентов и потерять 194 накопленные notes.

Одновременно нужна рабочая среда, где можно свободно писать документы, рисовать схемы, экспериментировать, коммитить промежуточные состояния — без страха что-то уронить.

## Варианты

### A. Работать прямо в `~/.openclaw/qoopia/` в отдельной ветке
- Плюсы: всё под рукой, один репо
- Минусы: риск случайно запустить команду на live-конфиге; история коммитов смешивается с watchdog-snapshot'ами; нет ментальной границы "это прод / это исследование"

### B. Создать отдельную репу `~/qoopia-v3/` только для docs/research/spec, код прод-Qoopia читаем read-only
- Плюсы: чёткая граница, невозможно случайно сломать прод, чистая история, легко поделиться/заархивировать
- Минусы: два места, нужно помнить где что лежит

### C. Делать всё в текущем worktree `interesting-moser`
- Плюсы: уже существует
- Минусы: worktree временный, при `ExitWorktree remove` всё потеряется; смешивается с openclaw work

## Решение

Выбран **Вариант B**.

## Последствия

- Прод-Qoopia невозможно сломать документами
- Чистая git-история: первый коммит = скелет workspace, дальше — только спека и research
- Когда дойдём до Фазы 5 (Execute), создадим отдельную ветку/репу для кода V3 — возможно `qoopia-v3/src/` внутри этой же репы, решим в Фазе 3
- Если проект замрёт, `~/qoopia-v3/` можно заархивировать и вернуться к нему позже не трогая прод
